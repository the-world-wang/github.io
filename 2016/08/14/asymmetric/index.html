不对称加密算法

不对称加密

之前一直搞不懂不对称加密是的流程，现在搞清楚了一点。相比于对称加密，不对称加密有公钥和私钥之分。公钥可以公开，私钥不能公开。
具体流程可如下：

  1. 甲生成公钥和私钥，并将公钥分享给乙
  2. 乙利用公钥加密信息，发送给甲
  3. 甲利用私钥解密得到明文
    甲向乙发送信息反之。

数字签名

一言以蔽之，数字签名就是将公钥密码反过来使用。签名者将讯息用私钥加密（这是一种反用，因为通常公钥密码中私钥用于解密），然后公布公钥;验证者使用公钥将加密讯息解密并比对消息（一般签名对象为消息的散列值。– 维基百科

摘要：对需要传输的文本，做一个HASH计算，一般采用SHA1，SHA2来获得。
签名：使用私钥对需要传输的文本的摘要进行加密，得到的密文即被称为该次传输过程的签名。
签名验证：数据接收端，拿到传输文本，但是需要确认该文本是否就是发送发出的内容，中途是否曾经被篡改。因此拿自己持有的公钥对签名进行解密（密钥对中的一种密钥加密的数据必定能使用另一种密钥解密。），得到了文本的摘要，然后使用与发送方同样的HASH算法计算摘要值，再与解密得到的摘要做对比，发现二者完全一致，则说明文本没有被篡改过。

SSH

SSH是一种网络协议，用于计算机之间登录。
早期计算机登录都是通过明文，比如FTP。一般我们常用的实现是OpenSSH

基本用法

基本流程

  1. 用户请求远程登录
  2. 远程主机将自己的公钥返回
  3. 用户输入密码后，通过公钥加密传递给远程主机
  4. 远程主机利用私钥解密，密码匹配则认证成功

口令登录

这是比较常用的登录方式。
用户第一次登录时，会提示是否需要记住远程主机的公钥，如果同意，则会将远程主机的公钥放置在
~/.ssh/known_hosts中

如果远程主机系统重装，则需要删除known_hosts中对应的记录。我遇到过2次，一次是更换了跳板机，另外一次remote换了一个系统。

公钥登录

口令登录比较麻烦，好在提供了公钥登录。

  1. ssh-keygen生成公钥
    会在~/.ssh文件夹下生成2个文件：id_rsa和id_rsa.pub，前者是私钥，后者是公钥。
  2. ssh-copy-id上传公钥
    上传完成后，公钥保存在./ssh/authorized_keys中

    在mac上ssh-copy-id命令没有，需要通过brew install ssh-copy-id安装（其实我们一般常用的SSH实现是OpenSSH）
